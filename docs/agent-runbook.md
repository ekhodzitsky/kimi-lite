# Runbook для автономного swarm-агента

Этот документ — расширенная инструкция для Kimi-агента, продолжающего работу над `kimi-lite` в режиме swarm. Промт в сессии задаёт цель и ссылается сюда; детали исполнения — в этом файле.

## 1. Цель и рамки

- Проект: `/Users/ekhodzitsky/Documents/personal/kimi-lite`.
- Цель: Go-порт https://github.com/MoonshotAI/kimi-code — паритет ключевых фич (функциональная эквивалентность UX, не копирование кода) и превосходство по надёжности/безопасности.
- Оригинал используется только как справочник по фичам. Не копируй реализацию, не импортируй код с несовместимой лицензией.

## 2. Источник правды

1. `TODO.md` — бэклог, волны, приоритеты, acceptance criteria.
   - `∥` — задачу можно выполнять параллельно с другими.
   - `→` — задачу нужно выполнять после предыдущей.
   - `depends_on` — явные зависимости; соблюдай строго.
   - Если `depends_on` ссылается на отсутствующий ID — проверь, не переименован/объединён ли он; если нашёл замену — обнови локальный `TODO.md` (не коммить) и продолжай; иначе эскалируй.
   - `priority table` задаёт внутриволновой порядок; номер волны определяется полем `Wave` задачи. Если таблицы нет — используй порядок перечисления задач в разделе волны.
   - При расхождении поля `Wave` и раздела `Execution waves` приоритет у поля `Wave`; если это создаёт неразрешимую коллизию — эскалируй.
   - Волна считается пустой, если в `TODO.md` нет задач с `Wave = N` в статусе, отличном от `done`.
2. `AGENTS.md` — целевая архитектура и конвенции. При расхождении с `TODO.md` приоритет у фактического бэклога.
3. `README.md` — пользовательская документация.
4. `CHANGELOG.md` — история изменений.
5. Этот файл (`docs/agent-runbook.md`) — workflow.

`TODO.md` читай и используй для планирования, но **не коммить**.

## 3. Перед началом работы

1. `git status` — проверь состояние рабочего дерева.
2. `git log --oneline -20` — посмотри недавнюю историю.
3. `make test -race` — baseline.
   - Если падает до твоих изменений (`baseline красный`) — эскалируй.
   - Если дерево грязное:
     - `git diff --name-only`.
     - Если изменения образуют завершённый инкремент текущей задачи, соответствуют acceptance criteria и проходят quality gates — создай `feature/<id>` и закоммить.
     - Иначе — эскалируй.
4. Изучи `TODO.md`, `AGENTS.md`, `README.md`, этот runbook.
   - Если `TODO.md` пуст или не содержит волн — эскалируй.
   - Пустые волны пропускай.

## 4. Порядок работы с волнами

- Не начинай волну N+1, пока не закрыта волна N.
- Внутри волны работай по `priority table`.
- Волна считается закрытой, когда:
  - все её задачи влиты в `main`/`integration`,
  - `make test -race` зелёный,
  - документация (ADR, CHANGELOG, README при необходимости) обновлена.
- Новые фичи бери только после закрытия текущей волны.
- Пустые волны не порождают `integration/wave-N`; используй номер следующей непустой волны.
- Для `depends_on` на отсутствующий или нереализованный ID — эскалируй.

## 5. Swarm-режим

- Используй `Agent`/`AgentSwarm`.
- Если swarm-инструмент недоступен — работай последовательно.
- Максимум **4 параллельных подагента**.
- Каждый подагент = **одна задача** с маркером `∥`.
- Зависимые задачи (`→`, `depends_on`) выполняй последовательно.
- Подагенту передавай:
  - ID задачи,
  - список файлов/пакетов,
  - acceptance criteria (из `TODO.md` или сформулированные на основе фичи),
  - имя ветки `feature/<id>`.
- Таймауты по сложности (`XS/S→S`, `M→M`, `L→L`):
  - S = 30 мин,
  - M = 60 мин,
  - L = 90 мин.
- Если подагент падает или зависает — перезапусти один раз; при повторе — эскалируй. Зависание — отсутствие ответа/прогресса в течение `2 × таймаут_задачи` (минимум 30 мин).
- Subagent пушит `feature/<id>` и сообщает финальный хеш.
- Родитель:
  1. Делает `git fetch origin`.
  2. Проверяет хеш в remote.
  3. Создаёт/обновляет `integration/wave-N` от актуального `main`.
  4. Мержит `feature/<id>` в `integration/wave-N`. Конфликты в `pkg/api`, `go.mod`, `go.sum`, CI/CD, `AGENTS.md`, лицензиях — эскалируй немедленно. Остальные конфликты пытайся разрешить в течение 15 минут; не получается — эскалируй.
  5. Перед merge запускает `make test -race`.
  6. Пушит `integration/wave-N` в origin.
  7. После зелёного CI на `integration/wave-N` открывает PR в `main`. Если CI падает — не открывай PR; локально воспроизведи падение и попробуй исправить в рамках задачи в течение 15 минут; если неясно или требует изменений вне задачи — эскалируй. Если CI не запускается/не настроен — убедись в локальном `make test -race` и отчитайся.
  8. Если PR недоступен (нет `gh`/веба) — оставляет ветку запушенной и отчитывается.

> Если несколько `∥`-задач трогают один и тот же пакет/интерфейс — сериализуй их или разбей на непересекающиеся области.
>
> В одиночном (не-swarm) режиме отчитывайся каждые 30 минут, даже если задача не завершена.

## 6. Git-правила

- Ветка: `feature/<id>`.
- Перед первым коммитом проверь `git config user.name`/`user.email`; при отсутствии установи разумные значения.
- Коммиты, ADR, README, CHANGELOG — **на английском**.
- Формат коммита: `<type/scope>: description (#<id>)` (например, `core: add context compressor (#C24)`). Допустимы `feat`, `fix`, `chore`, `docs`, `test`, `refactor`.
- Пушь после зелёного `make test -race` и по завершении задачи.
- **Force-push запрещён везде**, включая личные feature-ветки. Для обновления ветки используй merge из `integration/wave-N` или `main`.
- Не коммить артефакты сборки: `*.test`, `*.out`, `bin/*`, `.venv-yaml/`, `TODO.md`.

## 7. Quality gates

- Новая логика должна быть покрыта тестами.
- Багфиксы: регрессионный тест первым коммитом, когда возможно.
- Тесты — только с моками/`httptest`; **не используй реальные LLM/MCP** в юнит-тестах.
- После изменений запускай в порядке:
  1. `go mod tidy`
  2. `go vet ./...`
  3. `make test -race` (обязательно перед merge/пушем)
  4. `make lint`
  5. `make build`
- `make lint` обязателен перед merge.
  - Если падает на изменённых тобой файлах — исправь.
  - Если падает на нетронутом коде — эскалируй.
  - При проблеме с версией линтера запускай `golangci-lint` v2.12.2 напрямую.
- Smoke-test: `./bin/kimi-lite --help`.
- Флакующий тест: для затронутого пакета прогони `go test -race -count=10 ./<pkg>` (или конкретный тест, если известен); любое падение — подозрение на баг. Если воспроизводится — исправь. Если не чинится в разумный срок — откати и эскалируй.

## 8. Безопасность и ресурсы

- Файловый доступ — через `os.Root`/`ValidateFilePath`.
- HTTP — через `secureHTTPClient`/блокировку хостов.
- Явно отменяй `context`, закрывай каналы SSE, файловые дескрипторы sandbox и stdio MCP.
- Новые зависимости — только с разрешёнными лицензиями (MIT, BSD, Apache-2.0, ISC, MPL-2.0, CC0, Unlicense и другие permissive OSI-approved, не copyleft; **не GPL/AGPL/EPL/CDDL**). Перед `go get` проверь `LICENSE` в репозитории модуля (через Read/FetchURL/WebSearch).
- После `go get` запусти `govulncheck ./...`.

## 9. Документация

- ADR в `docs/adr/` и обновление `AGENTS.md` — при изменении public API, добавлении нового top-level пакета `internal/*`, смене sandbox-модели или протокола MCP.
- `CHANGELOG.md`/`README.md` — при новых пользовательских фичах или удалении фич из docs.
- Инфраструктурные/toolchain изменения (версия Go в `go.mod`, CI, линтер) — только после согласования.

## 10. `pkg/api`

- Сохраняй обратную совместимость.
- Новые интерфейсы/структуры/поля — разрешены.
- Изменение/удаление существующих сигнатур, добавление метода в существующий интерфейс, изменение семантики — **эскалация**, кроме задач из `TODO.md`, где acceptance criteria явно требуют breaking change; в таком случае сопроводи изменение ADR и обновлением `AGENTS.md`/`CHANGELOG`.

## 11. Эскалация

Остановись и сообщи:
- причину,
- текущую ветку,
- хеш коммита,
- состояние `make test`.

**Эскалируй**:
- baseline красный,
- `TODO.md` пуст/не валиден,
- `depends_on` на отсутствующий/нереализованный ID,
- конфликты merge, которые не удаётся разрешить за 15 минут,
- `make lint` падает на нетронутом коде,
- изменения лицензии, CI/CD (например, `.github/workflows/ci.yml`), toolchain,
- breaking changes в `pkg/api`,
- удаление фич вне `TODO.md`,
- runtime-зависимости (test-only зависимости, явно перечисленные в acceptance criteria задачи из `TODO.md`, разрешены),
- пустые волны между непустыми только если их пропуск нарушает ссылки в `CHANGELOG`/`ADR`/`TODO.md`; иначе применяй §4.

**Решай сам**:
- низкоуровневые баги (race, leak, panic) в затронутом коде — отдельным коммитом,
- крупный (>3 пакета) или несвязанный баг — добавь задачу в локальную копию `TODO.md` (не коммить) или в отдельный issue и эскалируй.

## 12. Задачи "implement or remove from docs"

- HIGH/MED — пытайся реализовать. Если не укладываешься в таймаут, требуется согласование UX или новые runtime-зависимости — эскалируй, а не удаляй.
- LOW/optional — удали.

## 13. Тестовая механика

Не меняй семантику проверяемого поведения. Изменение механики тестов (refactor, устранение флакинесса, адаптация к API) разрешено, если это указано в acceptance criteria.
