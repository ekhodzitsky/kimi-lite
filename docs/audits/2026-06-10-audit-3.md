## 🔴 P0 — Критические (5 штук)

### 1. SSRF redirect bypass — security theater (fake fix)
- **File:Line:** `internal/app/app.go:62-70`, `internal/core/tools.go:55-68`
- **Description:** `newSecureHTTPClient()` в `tools.go` содержит `CheckRedirect` с `isBlockedHost(req.URL.Hostname())`. НО `app.go:62-70` создаёт СВОЙ `&http.Client{}` с `CheckRedirect`, который проверяет только `len(via) >= 3` и **НЕ** вызывает `isBlockedHost`. Этот insecure client передаётся в `NewBuiltInToolExecutor`, где `if httpClient != nil` использует его напрямую. `newSecureHTTPClient()` никогда не вызывается в production path.
- **Impact:** Полный SSRF. `fetch_url` на внешний URL, который возвращает `Location: http://127.0.0.1:9999/secret` или `Location: http://169.254.169.254/latest/meta-data/`, следует редиректу.
- **Proof:** `app.go:62` → `httpClient` (no isBlockedHost) → `NewBuiltInToolExecutor` line 34-36: `if httpClient != nil { use it }`. `newSecureHTTPClient()` — dead code в runtime.
- **Recommendation:** Удалить custom client из `app.go`. Передавать `nil` в `NewBuiltInToolExecutor`, чтобы он использовал `newSecureHTTPClient()`.

### 2. Stale approval leak across turns
- **File:Line:** `internal/core/turn.go:37`, `75-81`, `170-176`
- **Description:** `approvalCh` buffered to 1. `RunTurn` (строка 85) не очищает канал перед запуском нового `run()`. Если предыдущий `run()` вышел по ошибке (ctx cancelled) во время approval pause, а `ResumeWithApproval` уже записал решение в канал — это решение остаётся в буфере. Новый `run()` доходит до `select { case approvals = <-tm.approvalCh }` и **немедленно применяет stale approval** к новому turn.
- **Impact:** Cross-turn state contamination. Approval для предыдущего turn автоматически применяется к следующему. Security boundary violation.
- **Recommendation:** Drain `approvalCh` в начале `RunTurn`, или закрывать и создавать новый канал per turn.

### 3. setError не очищает pendingCalls
- **File:Line:** `internal/core/turn.go:365-376`
- **Description:** `setError` обновляет `turn.State` и `turn.Error`, но не делает `tm.pendingCalls = nil`. Если ошибка произошла во время approval pause, `PendingApprovals()` продолжает возвращать tool calls из мёртвого turn.
- **Impact:** TUI показывает phantom tool calls. `ResumeWithApproval` может быть вызван против мёртвого turn.
- **Recommendation:** `tm.pendingCalls = nil` в `setError`, под `tm.pendingMu`.

### 4. Go version misalignment — Wave 2 fix не применился
- **File:Line:** `go.mod:3`, `.github/workflows/ci.yml:21`
- **Description:** `go.mod` требует `go 1.25.0`. CI workflow указывает `go: ['1.23']`. Агент Wave 2 (agent-30) заявил, что поменял на `go 1.23`, но файл на диске всё ещё `go 1.25.0`. Go toolchain с версии 1.21+ **отказывается собирать** модули с `go` directive выше своей версии.
- **Impact:** CI полностью сломан. Каждый push/PR будет падать с `toolchain mismatch`.
- **Recommendation:** Либо понизить `go.mod` до `go 1.23` (и запустить `go mod tidy`), либо поднять CI до `1.25` / `stable`.

### 5. 4 активные уязвимости в зависимостях
- **File:** `go.mod` (transitive)
- **Description:** `govulncheck ./...` находит 4 уязвимости с прямыми trace paths в код приложения:
  - **GO-2026-5039** (`net/textproto`): Arbitrary inputs в errors без escaping. Trace: `execFetchURL` → `io.ReadAll` → `textproto.Reader.ReadMIMEHeader`.
  - **GO-2026-5037** (`crypto/x509`): Inefficient hostname parsing. Trace: `main.go:59` → `fmt.Fprintf` → `x509.Certificate.Verify`.
  - **GO-2026-4971** (`net`): Panic на Windows с NUL byte. Trace: `execFetchURL` → `http.Client.Do` → `net.Dialer.DialContext`.
  - **GO-2026-4918** (`golang.org/x/net`): Infinite loop в HTTP/2 transport. Trace: `execFetchURL` через `http.Client`.
- **Impact:** DoS (infinite loops, panics), потенциальная утечка информации через error messages.
- **Recommendation:** Обновить Go toolchain до 1.26.4+, `go get golang.org/x/net@latest`, добавить `govulncheck` в CI как обязательный шаг.

## 🟠 P1 — Высокий приоритет (8 штук)

### 6. Error bodies с API keys утекают в SQLite
- **File:Line:** `internal/llm/client.go:248, 295, 302`, `internal/core/turn.go:368`, `internal/store/sqlite.go`
- **Description:** `doRequest` возвращает `fmt.Errorf("client error %d: %s", resp.StatusCode, string(respBody))`. Некоторые провайдеры возвращают stack traces, account IDs или echoed headers в error bodies. Этот error string попадает в `turn.Error`, который сохраняется в SQLite через `saveTurn`.
- **Impact:** API keys или account secrets могут быть записаны на диск в `~/.local/share/kimi-lite/sessions.db`.
- **Recommendation:** Sanitize error bodies перед сохранением. Не включать `respBody` в `turn.Error`.

### 7. MCP subprocess inherits full environment
- **File:Line:** `internal/mcp/transport.go:102-119`
- **Description:** `StdioTransport.Connect` создаёт `exec.Command` без установки `cmd.Env`. По умолчанию child наследует ВСЕ env vars родителя, включая `KIMI_LLM_API_KEY`, `MOONSHOT_API_KEY`, `HOME`, `SSH_AUTH_SOCK`.
- **Impact:** Если MCP binary или сервер скомпрометирован — полная эксфильтрация secrets.
- **Recommendation:** `cmd.Env = []string{"PATH=/usr/bin:/bin"}` (минимальный allowlist).

### 8. wordWrap ломает multi-byte UTF-8
- **File:Line:** `internal/tui/messages/messages.go:277-291`
- **Description:** `wordWrap` использует `len(line) > width` и `line[:width]` — байтовые операции. Multi-byte руны (emoji, CJK) разрезаются посередине, производя invalid UTF-8.
- **Impact:** Terminal emulators могут показывать replacement characters или крашиться.
- **Recommendation:** Использовать `[]rune(line)` или `uniseg`/`runewidth` для корректного подсчёта ширины.

### 9. TUI: resumeWithApprovals — data race из command goroutine
- **File:Line:** `internal/tui/model.go:763-773`
- **Description:** `resumeWithApprovals` возвращает `tea.Cmd`, которая мутирует `m.pendingApprovals`, `m.approvalIndex`, `m.approvalDecisions` из goroutine (Bubble Tea commands run async). `Update()` thread читает эти же поля.
- **Impact:** Verified data race. Race detector будет ловить это при стресс-тесте.
- **Recommendation:** Возвращать message и мутировать state только внутри `Update()`.

### 10. Input height hardcoded — layout ломается
- **File:Line:** `internal/tui/model.go:513, 918`
- **Description:** `inputHeight := 3` hardcoded в двух местах. `textarea.Model` auto-grows с контентом. При 5-line input textarea занимает 5 строк, но layout резервирует 3 — input перезаписывает status bar.
- **Impact:** Multi-line сообщения нечитаемы при вводе.
- **Recommendation:** Динамически вычислять `lipgloss.Height(m.input.View())`.

### 11. idgen — 0% coverage, нет тестов
- **File:Line:** `internal/idgen/idgen.go:13`
- **Description:** Пакет `idgen` генерирует IDs. Fallback на `time.Now().UnixNano()` при ошибке `crypto/rand`. Нет тестов на uniqueness, length, fallback path.
- **Impact:** При высокой конкуренции fallback может генерировать коллизии.
- **Recommendation:** Добавить `idgen_test.go` с тестами на uniqueness, length == 16, и fallback path.

### 12. System messages созданы без ID
- **File:Line:** `internal/app/app.go:194-209`
- **Description:** `AppendMessage` вызывается для git status и system messages без установки `msg.ID`. Нарушает инвариант, что все сообщения имеют ID.
- **Impact:** Пустые string IDs могут вызывать deduplication bugs или нарушать future schema constraints.
- **Recommendation:** `idgen.GenerateID()` для каждого system message.

### 13. O(n) rebuild on every stream chunk
- **File:Line:** `internal/tui/model.go:645-650`
- **Description:** `handleStreamChunk` вызывает `m.renderedContent.Reset()` и перестраивает `msg.View()` для **всех сообщений** на каждый chunk.
- **Impact:** Для длинных сессий streaming становится линейно медленнее.
- **Recommendation:** Append-only обновление `renderedContent` во время streaming.

## 🟡 P2 — Средний приоритет (6 штук)

### 14. TUI mouse focus без Focus/Blur
- **File:Line:** `internal/tui/model.go:497-521`
- **Description:** `handleMouseMsg` меняет `m.focused`, но не вызывает `m.input.Focus()` / `m.input.Blur()`. Курсор textarea продолжает мигать, даже когда focus на viewport.
- **Recommendation:** Вызывать Focus/Blur методы при смене focus через мышь.

### 15. t.Chdir в тестах — process-wide cwd mutation
- **File:Line:** `cmd/kimi-lite/main_test.go:94, 131, 171, 211`, `internal/config/config_test.go:83, 172`
- **Description:** `t.Chdir()` меняет process-wide working directory. Хотя `t.Parallel()` сейчас нет, это хрупкая практика. Если кто-то добавит `t.Parallel()` — получит race.
- **Recommendation:** Рефакторить код для приёма base path параметра вместо `os.Getwd()`.

### 16. Grep всё ещё следует за symlinks
- **File:Line:** `internal/core/tools.go:363-384`
- **Description:** Комментарий признаёт: "GNU grep -r follows symlinks. There is no simple portable flag to disable this."
- **Impact:** Information disclosure через symlink внутри sandbox → за пределы sandbox.
- **Recommendation:** Ручной обход дерева в Go вместо `grep -r`.

### 17. Stream errors невидимы для TUI
- **File:Line:** `internal/core/turn.go:286-316`, `internal/tui/model.go:628-636`
- **Description:** `consumeStream` вызывает `setError()`, `run()` выходит, `outCh` закрывается. TUI видит только `Done: true` и переходит в `TurnIdle`. `turn.Error` никогда не передаётся через `outCh`.
- **Impact:** Пользователь не видит ошибок сети/timeout. Думает, что turn завершился успешно.
- **Recommendation:** Передавать error через `outCh` или dedicated error channel.

### 18. Нет guard на максимальное число tool rounds
- **File:Line:** `internal/core/turn.go:166-258`
- **Description:** `for len(toolCalls) > 0` может зациклиться, если LLM входит в repetitive loop.
- **Impact:** Бесконечное потребление tokens, UI lockup.
- **Recommendation:** `maxToolRounds` константа (например, 10).

### 19. No integration tests, tests/integration/ пуст
- **File:Line:** `tests/integration/`
- **Description:** Директория существует, но пуста. AGENTS.md и README.md утверждают, что там E2E тесты.
- **Recommendation:** Либо написать integration tests, либо удалить директорию и обновить документацию.

## Общий вердикт

**Score: 4.9/10**

Проект прошёл две волны исправлений и ~50 фиксов, но **критические уязвимости остаются**, а некоторые "исправления" оказались security theater. Главная проблема — SSRF redirect bypass в `app.go`: код выглядит безопасным (`newSecureHTTPClient` с `isBlockedHost`), но production path его обходит полностью. Это классический пример "passes unit tests, fails in production".

Вторая критическая проблема — stale approval leak: buffering `approvalCh` исправил deadlock, но создал cross-turn contamination. Архитектура turn manager требует редизайна channel lifecycle.

TUI layer имеет реальные data races (`resumeWithApprovals`), hardcoded layout constants, и UTF-8 corruption. Тестовое покрытие обманывает: 75% в среднем, но `app.go` 6%, `idgen` 0%, и ключевые error paths не покрыты.

CI/CD сломан из-за go.mod mismatch. 4 CVE в dependencies. Рабочая директория загрязнена `.venv-yaml/`, `coverage.out`, `bin/`.

**Прежде чем проект можно будет считать production-ready, необходимо:**
1. Исправить SSRF redirect bypass в `app.go` (P0 #1)
2. Переделать lifecycle `approvalCh` — drain per turn (P0 #2)
3. Очистить `pendingCalls` в `setError` (P0 #3)
4. Выровнять Go версии — либо понизить go.mod, либо поднять CI (P0 #4)
5. Обновить Go toolchain и зависимости (P0 #5)
6. Sanitize error bodies перед persistence (P1 #6)
7. Очистить env для MCP subprocess (P1 #7)
8. Исправить wordWrap для UTF-8 (P1 #8)
9. Исправить data race в `resumeWithApprovals` (P1 #9)
10. Динамический input height (P1 #10)
