# Design: Port kimi-code 0.15.0/0.14.x parity features to kimi-lite

## Context

Upstream `MoonshotAI/kimi-code` released `0.15.0` (2026-06-15) with a set of features that improve UX, MCP compatibility, and robustness. `kimi-lite` is a Go reimplementation aiming for functional parity, so we port the most valuable subset.

## Goals

1. Close the MCP transport gap by adding **legacy SSE MCP server support**.
2. Add the **Interrupt hook event** so hooks can observe user interruption.
3. Preserve **image outputs from tools** in OpenAI-compatible chat completions.
4. **Recover resumed sessions** when an interrupted tool-call result was not recorded.
5. Add an **all-sessions picker** with search and paginated browsing.

## Non-goals

- Pixel-perfect UX replication of upstream TUI.
- Porting upstream skills/marketplace/registry changes unrelated to core parity.
- Breaking existing CLI/config compatibility.

## 1. Legacy SSE MCP transport

### Current state

`internal/mcp` supports:
- `StdioTransport` — JSON-RPC over stdio to a subprocess.
- `HTTPTransport` — JSON-RPC over HTTP POST to a streamable HTTP endpoint.

### Upstream behavior

Legacy MCP servers use Server-Sent Events (SSE): the client connects to an SSE endpoint, receives an endpoint URL for POSTing messages, and reads responses as SSE events.

### Design

- Add `internal/mcp/sse_transport.go` implementing `mcp.Transport`.
- Configuration: extend `api.MCPServerConfig` with `transport = "sse"` and `url`.
- Flow:
  1. `Connect()` opens a long-lived GET request to `{url}/sse` with `Accept: text/event-stream`.
  2. Parse SSE events; look for `endpoint` event containing the POST URL.
  3. `Send()` POSTs JSON-RPC requests to that endpoint.
  4. Responses arrive as SSE `message` events and are matched by request id.
- Reuse existing JSON-RPC framing (`mcp.JSONRPCRequest`, `JSONRPCResponse`).
- Use `netutil.SecureHTTPClient()` for SSRF-hardened outbound requests.
- `Close()` cancels the SSE request context and drains the response body.

### Tests

- Mock SSE server in `transport_sse_test.go`.
- Test endpoint discovery, request/response round-trip, and close behavior.

## 2. Interrupt hook event

### Current state

`pkg/api/types.go` defines `HookEvent` with `session_start`, `session_end`, `turn_start`, `turn_end`, `tool_call`, `tool_result`, `approval_request`, `approval_decision`.

### Upstream behavior

A new `interrupt` hook event fires when the user interrupts a running turn (e.g. Esc / Ctrl-C).

### Design

- Add `HookTurnInterrupt HookEvent = "turn_interrupt"` to `pkg/api/types.go`.
- In `internal/core/turn.go`, when `CancelAll` or stream interruption is triggered, call `tm.runHooks(ctx, api.HookTurnInterrupt, sessionID, turn.ID, "")`.
- Update `internal/core/hooks/runner.go` if it validates allowed events.
- Update prompt/system text to mention the new event.

### Tests

- Unit test that interruption triggers the hook.
- Hook runner test for event validation.

## 3. Image output preservation

### Current state

`api.Message.Content` is a `string`. Tool results are serialized into string content. There is no first-class image/multi-modal content.

### Upstream behavior

When a tool returns an image (e.g. screenshot, chart), it is preserved as an image content part in the assistant message when using OpenAI-compatible chat completions.

### Design

- Extend `api.Message` to support multi-modal content:
  - Add `ContentParts []ContentPart` or similar.
  - Keep `Content string` for backward compatibility.
  - A `ContentPart` has `Type` (`text`, `image_url`) and fields for text or image URL/base64 data.
- Update `internal/llm` request builders to emit OpenAI `content` arrays when parts are present.
- Update tool result rendering so image outputs can be surfaced by tools and stored in messages.
- Update token estimator to count image tokens (rough heuristic).

### Tests

- Message serialization/deserialization round-trip.
- LLM client request body includes image content.
- Compressor and exporter handle multi-modal messages.

## 4. Session recovery for interrupted tool calls

### Current state

Turns store messages and turn state. If the process dies after a tool call was dispatched but before the result was recorded, resuming the session may leave a dangling `tool_call` assistant message without a matching `tool` result.

### Upstream behavior

On resume, detect dangling tool calls and either:
- Re-execute them, or
- Mark them as failed/cancelled so the turn can continue.

### Design

- In `internal/core/session.go`/`turn.go`, when loading a turn, validate message sequence: every `assistant` message with `ToolCalls` must have corresponding `tool` results.
- If missing, emit synthetic tool results with an error/cancelled state.
- Add a `TurnState` value for `TurnRecovering` if needed.
- Expose a recovery hook event `HookTurnRecover` (optional).

### Tests

- Store fixture with incomplete tool-call history.
- Turn resume test verifies synthetic results are appended.

## 5. All-sessions picker

### Current state

TUI has `/sessions` command that lists sessions in the current working directory. `internal/store.ListSessions` supports path-based listing.

### Upstream behavior

A picker view shows all sessions across working directories with:
- Name search.
- Paginated browsing.
- Clipboard-ready resume commands.
- Toggle between current-dir and all-sessions modes.

### Design

- Add `internal/store.ListAllSessions(ctx, limit, offset)` or extend `ListSessions` with a sentinel path (e.g. `""` or `"*"`) to list sessions regardless of path.
- Extend TUI `/sessions` overlay:
  - Add search input.
  - Add pagination (PageUp/PageDown).
  - Show working directory for each session.
  - Add toggle `a` for all/current dir.
- When a session from another directory is selected, change the working directory or start a resume command.

### Tests

- Store test for listing sessions across paths.
- TUI command test for picker interactions.

## Integration and sequencing

Recommended implementation order:

1. Interrupt hook event — smallest, establishes pattern.
2. Legacy SSE MCP transport — self-contained, high value.
3. Session recovery for interrupted tool calls — improves robustness.
4. Image output preservation — broader API change; do after recovery.
5. All-sessions picker — largest TUI feature; do last.

## Risks and mitigations

| Risk | Mitigation |
|------|------------|
| Multi-modal `Message` change breaks store schema | Add migration for new columns; keep `Content` as fallback. |
| SSE transport introduces new concurrency patterns | Reuse HTTP client patterns; add race tests. |
| All-sessions picker duplicates `/sessions` logic | Refactor existing picker into reusable component. |
| Recovery changes turn state machine | Add explicit `TurnRecovering` state and tests. |

## Success criteria

- All five features have unit tests.
- `go test -race ./...` passes.
- `golangci-lint run ./...` reports zero issues.
- CHANGELOG.md is updated under `[Unreleased]`.
